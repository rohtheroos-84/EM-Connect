# EM-Connect

EM-Connect is a backend-first event management system i built to learn real-world system design with spring boot, golang workers, and a react frontend connected through rabbitmq.

it now supports event lifecycle management, registration concurrency handling, async ticket generation, rich email flows (including reminders and password reset), real-time websocket updates, analytics, and a bauhaus-styled ui.

## what this project is

an event management platform where:

- spring boot handles domain logic, rest apis, auth, scheduling, and database transactions
- three golang services handle async work — notifications, ticket generation, and websocket fan-out
- rabbitmq keeps services loosely coupled through domain events
- a react frontend consumes rest + websocket streams in real time
- docker compose brings up the local infra stack

this started as a learning project and grew into a production-style multi-service system.

## core features

### authentication
- user registration and login with email/password
- google oauth2 login with account linking
- forgot password flow with verification code (request code, verify code, reset password)
- jwt-based stateless auth (24h expiry)
- role-based access control with `user` and `admin`
- bcrypt password hashing
- protected routes + frontend auth context

### event management
- full crud: create, update, publish, cancel, complete
- event state machine: draft -> published -> cancelled/completed
- only published events are visible publicly
- category support (`technology`, `social`, `sports`, `music`, `education`, `business`, `health`, `art`, `other`)
- tag-based discovery and filtering
- banner upload (jpeg/png/gif/webp)
- deterministic bauhaus svg fallback banner generation when no upload exists
- search + pagination + category/tag filters
- organizer-owned event management flows

### registrations
- register for published events
- atomic capacity enforcement using pessimistic locking
- duplicate registration prevention
- cancellation and status tracking
- concurrency-safe behavior under heavy parallel requests

### tickets
- async ticket generation after confirmation events
- qr codes generated by ticket-worker
- signed payload + json metadata storage
- ticket retrieval, qr download, and validation
- check-in support with idempotent validation behavior

### notifications
- registration confirmed/cancelled emails
- event published/cancelled emails
- scheduled reminder emails (24h + 1h)
- welcome/login/password-changed/check-in/password-reset emails
- sendgrid-based delivery with retry + dlq handling
- local mailhog still available for dev-level smtp inspection

### real-time updates
- websocket hub in go with topic-based subscriptions
- live participant count updates
- live event announcements
- frontend auto-reconnect and listener-based update handling

### user profiles
- edit profile info
- change password
- avatar upload
- registration history and account stats

### calendar export
- export `.ics` files
- one-click google calendar links
- one-click outlook calendar links

### analytics + admin
- analytics dashboard with recharts (trends, capacity, activity, top events/locations)
- admin panel with overview, event management, and user management
- role management and system-level operations

### ui system
- bauhaus-inspired visual language
- geometric accents, hard shadows, bold primary palette
- dark mode with persisted preference

## architecture overview

```
client (react)
    |
    |- rest api ---> spring boot api ---> postgresql
    |                  |
    |                  |- publishes domain events ---> rabbitmq (topic exchange)
    |                                              |
    |                        +---------------------+---------------------+
    |                        |                     |                     |
    |                        v                     v                     v
    |                 notification-worker     ticket-worker        websocket-hub
    |                        |                     |                     |
    |                        v                     v                     |
    |                  sendgrid/mailhog      qr + metadata files         |
    |                                                                    |
    +----------------- websocket ----------------------------------------+
```

production note (free tier): notification and ticket processors are deployed as render web services (not background workers), each exposing a lightweight `/health` endpoint for port binding.

keep-alive monitoring:
- uptime robot monitors all 4 render services every 5 minutes.
- monitor set: backend api, websocket hub, notification worker, ticket worker.
- status page: https://stats.uptimerobot.com/v6aGZHL957

- spring boot api: core domain logic + auth + persistence + event publishing
- notification-worker (go): consumes `registration.*`, `event.*`, `user.*` and sends html mail
- ticket-worker (go): consumes `registration.confirmed` and generates qr tickets
- websocket-hub (go): consumes registration/event broadcasts and pushes real-time updates
- postgresql 16: primary datastore with flyway migrations (`v1` to `v10`)
- rabbitmq 3.13: `em.events` exchange + `em.events.dlx` dead-letter exchange
- mailhog: optional local mail sink via docker profile `dev`

## running locally

### prerequisites
- docker + docker compose
- java 17+
- go
- node.js 18+ + npm

### 1) start infrastructure

```bash
docker compose --profile dev up -d
```

`--profile dev` starts mailhog too. if you don't need mailhog, run plain `docker compose up -d`.

### 2) start backend api

```bash
cd services/api
./mvnw spring-boot:run
```

windows powershell alternative:

```powershell
cd services/api
.\mvnw.cmd spring-boot:run
```

### 3) start go workers (separate terminals)

```bash
cd services/notification-worker
go run main.go
```

```bash
cd services/ticket-worker
go run main.go
```

```bash
cd services/websocket-hub
go run main.go
```

### 4) start frontend

```bash
cd frontend
npm install
npm run dev
```

### local urls
- frontend: `http://localhost:3000`
- api: `http://localhost:8080`
- rabbitmq ui: `http://localhost:15672` (`emconnect` / `emconnect`)
- mailhog ui (when dev profile is used): `http://localhost:8025`

### local/prod environment switching
- keep production secrets only in render/netlify env settings.
- for local runs, use default local values (docker postgres/rabbitmq) and do not export cloud env vars in your shell.
- if you need to test cloud resources locally, use a separate terminal profile/session so you can switch back cleanly.

## messaging topology

- exchange: `em.events` (topic)
- dead-letter exchange: `em.events.dlx` (topic)
- queues:
  - `notification.queue` bound to `registration.*`, `event.*`, `user.*`
  - `ticket.queue` bound to `registration.confirmed`
  - `websocket.queue` bound to `registration.*`, `event.*`
- dead-letter queue: `em.events.dlq`

event groups currently published include:
- registration events (`confirmed`, `cancelled`, `checkedin`)
- event events (`published`, `cancelled`, `updated`, `reminder`)
- user events (`registered`, `login`, `password_changed`, `password_reset`)

### free-tier caveats

- render web services spin down on inactivity (cold starts can add delay).
- websocket and async processing responsiveness can degrade after idle periods.
- if you need steadier queue-processing behavior on free tier, keep worker `/health` endpoints warm with an external uptime monitor.
